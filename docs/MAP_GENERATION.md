# Генерация случайных карт — проектный документ

## Цель

Добавить возможность генерировать случайные игровые карты с настраиваемыми параметрами. Карты должны быть играбельными, эстетичными и сбалансированными.

---

## Варианты алгоритмов

### 1. Клеточные автоматы (Cellular Automata)

**Принцип:** Начинаем с хаотического заполнения (каждая клетка случайно суша/вода с заданной вероятностью), затем несколько итераций применяем правила сглаживания.

**Алгоритм:**
```
1. Заполнить сетку случайно: P(суша) = 45-55%
2. Повторить 4-6 раз:
   Для каждой клетки:
     Посчитать N = количество соседей-суши (в радиусе 1, включая диагонали, итого 8)
     Если N >= 5 → клетка становится сушей
     Если N <= 3 → клетка становится водой
     Иначе → остаётся как есть
3. Край карты (2 клетки от границы) — принудительно вода
```

**Плюсы:**
- Естественные формы островов, похожие на реальную географию
- Гладкие береговые линии без "шахматного" паттерна
- Параметр начальной плотности напрямую влияет на размер острова
- Простая реализация (~50 строк)

**Минусы:**
- Может создавать изолированные острова (нужна проверка связности)
- Формы непредсказуемы — иногда слишком длинные "змейки"
- Нет гарантии симметрии (для этого нужна обёртка)

**Вариации:**
- **Начальное зерно:** вместо полностью случайного заполнения, можно начать с круга или эллипса суши в центре, затем применить CA для органичных краёв
- **Два прохода:** первый с жёсткими правилами (5/3) для крупных форм, второй с мягкими (4/4) для сглаживания
- **Эрозия:** финальный проход удаляющий одинокие клетки суши (меньше 2 соседей) для чистоты

---

### 2. Шум Перлина / Simplex Noise

**Принцип:** Генерируем 2D шум, порог отсечения определяет суша/вода.

**Алгоритм:**
```
1. Для каждой клетки (r,c) вычислить noise(r*scale, c*scale)
2. Если noise > threshold → суша
3. Threshold регулирует плотность (0.0 = 50% суши, +0.2 = меньше суши)
4. Scale регулирует "масштаб" ландшафта (больше = крупнее формы)
```

**Плюсы:**
- Контролируемый масштаб форм
- Детерминистичный при одном seed
- Гладкие формы

**Минусы:**
- Нужна реализация Perlin/Simplex noise (или подключение библиотеки)
- Формы часто "облачные", менее островные чем CA
- Порог отсечения создаёт резкие границы (можно смягчить фильтром)

---

### 3. Random Walk / Drunkard's Walk

**Принцип:** Начинаем из центра, "пьяный" агент случайно ходит и превращает клетки в сушу.

**Алгоритм:**
```
1. Начать из центра карты
2. Повторить N раз (N = целевое количество суши):
   Сделать случайный шаг в одном из 4 направлений
   Текущая клетка → суша
3. Опционально: запустить несколько агентов из разных точек
```

**Плюсы:**
- Гарантированная связность (агент всегда идёт по связному пути)
- Очень простая реализация
- Создаёт "органичные" формы с пещерами и коридорами

**Минусы:**
- Формы часто слишком вытянутые/"змеиные"
- Мало контроля над итоговой формой
- Плохо подходит для создания компактных островов

---

### 4. BSP / Room-based (для архипелагов)

**Принцип:** Разделить карту на регионы, в каждом генерировать мини-остров, соединить мостами.

**Алгоритм:**
```
1. Разделить карту на 2-4 региона (рекурсивное BSP-деление)
2. В каждом регионе: сгенерировать остров (CA или эллипс)
3. Соединить соседние острова перешейками/мостами (1-2 клетки)
4. Разместить корабли на разных островах
```

**Плюсы:**
- Архипелаги — самый интересный тип карт для стратегии
- Мосты создают "бутылочные горлышки" — ключевые точки контроля
- Каждый остров — своя мини-территория

**Минусы:**
- Более сложная реализация
- Нужно гарантировать что мосты проходимы
- Баланс расстояний между островами

## Симметрия

### Зеркальная генерация (для сбалансированных игр)

**2-fold симметрия (дуэль, 2 игрока):**
```
1. Генерировать только ВЕРХНЮЮ половину карты (строки 0-6)
2. Отзеркалить по горизонтали на нижнюю (строки 7-12)
3. Корабли: один сверху, один снизу — зеркально
```
Это гарантирует что оба игрока в абсолютно одинаковых условиях.

**4-fold симметрия (4 игрока):**
```
1. Генерировать только ВЕРХНИЙ-ЛЕВЫЙ квадрант (строки 0-6, столбцы 0-6)
2. Отзеркалить по горизонтали → верхний-правый
3. Отзеркалить всё верхнее по вертикали → нижнее
4. Корабли по 4 сторонам: зеркально
```
Абсолютная симметрия для 4 игроков. Каждый стартует в одинаковых условиях.

**Вращательная симметрия (элегантнее 4-fold):**
```
1. Генерировать один квадрант
2. Повернуть на 90°, 180°, 270° для остальных трёх
3. Результат: карта с вращательной (но не зеркальной) симметрией
```
Более интересно визуально — карта не выглядит как "обои".

### Незеркальная генерация (для разнообразия)

**Когда симметрия не нужна:**
- Однопользовательская игра vs ИИ — баланс менее критичен
- "Хардкор" режим — случайный остров, кому как повезёт
- Кооперативный режим (если добавится)

**Параметры для незеркальных карт:**
- Начальная плотность: 30-70%
- Количество итераций CA: 3-7
- Минимальный размер острова (отсечь слишком маленькие)
- Скалы: случайные вкрапления (1-5%) на суше

## Подводные камни и решения

### 1. Связность острова
**Проблема:** CA может создать несколько изолированных кусков суши. Пираты не смогут перейти с одного на другой.

**Решение:** После генерации — BFS/flood fill из стартовой позиции. Если не все клетки суши достижимы:
- Вариант А: Оставить только самый большой связный компонент, удалить мелкие
- Вариант В: Соединить компоненты мостами (кратчайший путь через воду)
- Вариант С: Перегенерировать

**Рекомендация:** Вариант А (простой, надёжный) + проверка что оставшийся остров достаточно большой.

### 2. Размещение кораблей
**Проблема:** Корабль должен быть на водной клетке, смежной с сушей. Для симметричных карт — на симметричных позициях.

**Решение:**
```
1. Найти все водные клетки смежные с сушей
2. Отфильтровать: только на краю карты (ряды 0,12 или столбцы 0,12)
   или хотя бы в "открытой воде" (не в озере внутри острова)
3. Для симметрии: выбрать одну позицию, зеркалить для остальных
4. Проверить что от каждого корабля можно дойти до суши
```

### 3. Скалы блокируют путь
**Проблема:** Случайные скалы могут создать непроходимую стену.

**Решение:** 
- Генерировать скалы ПОСЛЕ проверки связности суши
- Каждую скалу проверять: если её добавление разрывает связность → не добавлять
- Или: скалы только в "толстых" местах (соседей-суши >= 5)

### 4. Слишком мало/много суши
**Проблема:** Для 100 тайлов колоды нужно минимум ~100 клеток суши. Если генератор создаёт остров из 30 клеток — тайлы не поместятся.

**Решение:**
- Минимальный порог: `landCount >= deckSize` (обычно ~100)
- Если не достигнуто — увеличить начальную плотность и перегенерировать
- Максимум 3-5 попыток, потом fallback на classic

### 5. Озёра внутри острова
**Проблема:** CA может создать "дырки" в суше — внутренние озёра. Это не обязательно плохо (красиво), но пираты могут застрять если упадут в воду внутри острова.

**Решение:**
- Вариант А: Заполнить внутренние озёра сушей (flood fill от краёв → всё что не достигнуто = внутреннее озеро)
- Вариант Б: Оставить озёра — они добавляют интерес. Пираты могут плавать.
- **Рекомендация:** Оставить маленькие озёра (1-3 клетки), заполнить большие.

### 6. Эстетика
**Проблема:** Случайные карты могут выглядеть "невзрачно" — нет характера.

**Решения:**
- Дать карте случайное название из пула ("Остров Черепов", "Затерянный архипелаг", ...)
- Добавить визуальный превью перед стартом
- Показывать статистику (размер, кол-во суши, симметрия)

## Рекомендуемый план реализации

### Фаза 1: Минимальный генератор (CA + связность)
- Клеточный автомат с начальной плотностью
- Проверка связности (BFS)
- Автоматическое размещение кораблей
- Кнопка "Случайная карта" в меню

### Фаза 2: Симметрия
- 2-fold зеркальная для дуэли
- 4-fold/вращательная для 4 игроков
- Параметр выбора в GUI

### Фаза 3: Расширенные типы
- Архипелаги (BSP + мосты)
- Скалы с проверкой проходимости
- Озёра (заполнение внутренних)

### Фаза 4: Полировка
- Превью карты перед стартом
- Случайные названия
- Сохранение понравившейся случайной карты через редактор

## Оценка сложности

| Фаза | Строк кода | Время |
|-------|-----------|-------|
| 1. CA + BFS + кнопка | ~150 C++ + ~30 QML | 1-2 часа |
| 2. Симметрия | ~80 C++ + ~20 QML | 1 час |
| 3. Архипелаги + скалы | ~200 C++ | 2-3 часа |
| 4. Полировка | ~50 C++ + ~100 QML | 1-2 часа |

**Итого: ~500 строк кода, 5-8 часов работы.**

Основной риск — не алгоритм (он простой), а отладка граничных случаев: слишком маленькие острова, неразмещаемые корабли, непроходимые скалы. Для этого sandbox-режим будет очень полезен.
